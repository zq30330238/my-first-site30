# 架构规划技能

在写代码之前先建规划文件，确保思路清晰、步骤明确。

## 触发条件

以下场景自动调用：
- 新项目/新功能启动
- 用户说"规划"、"设计方案"、"梳理架构"
- 任务涉及 3+ 个文件修改
- 用户不确定怎么下手

## 规划文件结构

在项目根目录 `.claude/plans/` 下创建规划文件，命名格式：`{slug}-plan.md`

### 规划文件必须包含

```markdown
# {项目名称}

## 1. 目标
- 要解决什么问题
- 成功标准是什么

## 2. 架构概览
- 技术栈
- 目录结构
- 模块划分

## 3. 关键文件
| 文件 | 作用 | 改动类型 |
|------|------|----------|
| xxx  | xxx  | 新建/修改/删除 |

## 4. 数据流
输入 → 处理 → 输出 的完整链路

## 5. 实施步骤
1. [ ] 第一步
2. [ ] 第二步
...

## 6. 风险点
- 可能出现的问题及应对
```

## 规划原则

1. **先想清楚再动手**：规划文件是思考过程的载体
2. **承认不确定性**：不确定的地方标 `[待确认]`，主动问用户
3. **逐步细化**：先写框架，再逐个模块展开
4. **保持更新**：执行中遇到变化，及时更新规划文件
5. **简洁优先**：规划文件不是论文，够用就行

## 输出方式

1. 创建 `.claude/plans/{slug}.md` 文件
2. 向用户简述规划要点
3. 等用户确认后再开始写代码

## 适用场景
- 新功能开发
- 重构项目结构
- 多文件修改
- 跨模块联调
- 任何"不知道从哪下手"的任务
